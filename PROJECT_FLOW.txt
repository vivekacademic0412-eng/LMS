LMSAM PROJECT CODE FLOW, STRUCTURE, AND FUNCTIONALITY
Generated from the current codebase in this repository.

================================================================
1. PROJECT SUMMARY
================================================================

This project is a Laravel 12 based Learning Management System (LMS).
It supports:

- role-based login and access control
- admin and superadmin management tools
- manager/HR reporting and exports
- IT monitoring and audit visibility
- trainer workflows for assigned learners
- student course access, progress, submissions, and certificates
- demo user workflows with demo tasks and demo videos
- secure media delivery for course files using Cloudinary
- in-app notifications
- activity logging and audit tracking

Main project identity:

- Backend framework: Laravel 12
- PHP version: 8.2+
- Frontend build: Vite
- Styling: Tailwind CSS v4 plus large custom Blade/CSS styling
- PDF generation: barryvdh/laravel-dompdf
- Secure media integration: Cloudinary PHP SDK

================================================================
3. HIGH LEVEL REQUEST FLOW
================================================================

Normal request flow:

1. Browser sends request to Laravel through public/index.php.
2. Laravel boots application from bootstrap/app.php.
3. Route is matched in routes/web.php.
4. Middleware runs before controller logic:
   - guest/auth
   - active user check
   - secure headers
   - activity logging for write actions
   - role check for restricted sections
5. Controller validates request.
6. Controller loads or updates Eloquent models.
7. Services are used where business logic is shared:
   - ActivityLogger
   - StudentCertificateService
   - CloudinaryPrivateMediaService
8. Controller returns Blade view, file response, redirect, streamed response, or download.
9. SecurityHeadersMiddleware adds CSP and other response headers.
10. LogActivity stores audit records after successful POST/PUT/PATCH/DELETE requests.

4. ACCESS CONTROL AND ROLES
================================================================

Roles are defined in app/Models/User.php:

- superadmin
- admin
- manager_hr
- it
- trainer
- student
- demo

Role usage:

- superadmin
  Full access to all management areas.

- admin
  Operational management access, but not allowed to manage superadmin/admin users.

- manager_hr
  View-oriented reporting access and HR export features.

- it
  View-oriented system monitoring and audit visibility.

- trainer
  Can see assigned students, assigned courses, submission queues, and quiz-live tools.

- student
  Can access enrolled courses, submit course work, and download earned certificates.

- demo
  Can access assigned demo tasks and demo videos and submit demo task responses.

Middleware involved:

- app/Http/Middleware/RoleMiddleware.php
  Blocks access if user role is not one of the allowed route roles.

- app/Http/Middleware/ActiveUserMiddleware.php
  Immediately logs out inactive users and sends them back to login.

- app/Http/Middleware/SecurityHeadersMiddleware.php
  Adds CSP, frame, cache, and browser hardening headers.

- app/Http/Middleware/LogActivity.php
  Logs successful write actions to activity_logs.

10. KEY FUNCTIONAL FLOWS
================================================================

10.1 Authentication flow

Files:

- routes/web.php
- app/Http/Controllers/AuthController.php
- app/Http/Middleware/ActiveUserMiddleware.php

Flow:

1. User opens /login.
2. AuthController@showLogin returns login view unless already authenticated.
3. POST /login validates email and password.
4. RateLimiter blocks repeated failures after 5 attempts.
5. Auth::attempt checks credentials.
6. Inactive users are logged out immediately and a blocked_login activity is saved.
7. Successful login regenerates session and saves login activity.
8. User is redirected to dashboard.


10.2 Dashboard flow

Files:

- app/Http/Controllers/DashboardController.php
- resources/views/dashboard/*

Flow:

1. User opens /dashboard.
2. Controller checks active user.
3. Controller resolves dashboard mode:
   - student
   - trainer
   - demo
   - admin
   - viewer
4. Shared stats are calculated.
5. Role-specific collections are built:
   - student learning items, pending tasks, recent submissions, certificates
   - trainer stats and counts
   - demo assignments, videos, categories, recent admin demo submissions
   - admin summary cards and shortcuts
6. Blade dashboard renders role-specific partials.


10.3 User management flow

Files:

- app/Http/Controllers/UserManagementController.php
- resources/views/users/index.blade.php

Flow:

1. Admin opens /users.
2. Users are filtered by role/status.
3. Creating a user:
   - validates role against current actor permissions
   - optionally uploads avatar
   - hashes password
   - sends welcome/access email
4. Updating a user:
   - checks permission against target role
   - updates profile data, role, active state, avatar, and optional password
5. Deleting a user:
   - cannot delete self
   - admin cannot delete admin/superadmin


10.4 Course and category management flow

Files:

- app/Http/Controllers/CourseCategoryController.php
- app/Http/Controllers/CourseController.php
- resources/views/courses/*

Flow:

1. Admin/HR/IT/trainer can view catalog pages.
2. Only admin/superadmin can create/update/delete categories and courses.
3. Category thumbnails and course thumbnails are stored on public disk.
4. Courses belong to a category and optional subcategory.
5. Course titles are unique inside category/subcategory combinations.
6. Trainers can only open courses they are assigned to through enrollments.


10.5 Course builder flow

Files:

- app/Http/Controllers/CourseWeekController.php
- app/Models/CourseWeek.php
- app/Models/CourseSession.php
- app/Models/CourseSessionItem.php

Flow:

1. Admin creates a week under a course.
2. Admin creates a session under a week.
3. System auto-generates 4 session items:
   - Intro PPT / Video
   - Main Video
   - Task
   - Quiz
4. Admin updates session items with:
   - title
   - resource type
   - content
   - external URL
   - secure uploaded file
5. If file is uploaded:
   - CloudinaryPrivateMediaService uploads file
   - old Cloudinary asset is deleted if replaced
   - Cloudinary metadata is saved on item


10.6 Enrollment flow

Files:

- app/Http/Controllers/CourseEnrollmentController.php
- resources/views/enrollments/index.blade.php

Flow:

1. Admin assigns a student to a course, optionally with a trainer.
2. Existing enrollment is reused if same student/course already exists.
3. assigned_by is stored.
4. Email is sent to student with course access information.
5. Database notification is also sent if notifications table exists.
6. Student later sees course inside My Courses.


10.7 Student learning flow

Files:

- app/Http/Controllers/CourseEnrollmentController.php
- app/Http/Controllers/CourseMediaController.php
- app/Http/Controllers/CourseItemSubmissionController.php
- resources/views/student/*

Flow:

1. Student opens /my-courses and sees enrolled course IDs.
2. Student opens /my-courses/{course}.
3. System verifies enrollment ownership.
4. System ensures CourseProgress rows exist for all session items.
5. Student opens content items.
6. CourseMediaController marks the viewed item completed in course_progress.
7. Pending next item is calculated for resume behavior.
8. Student submits tasks/files or quiz text answers.
9. Submission creates CourseItemSubmission and updates progress.
10. Trainer/admin reviews submission.
11. If reviewed, progress stays completed.
12. If revision requested, progress is reset to incomplete.
13. When all items are completed, certificate becomes available.


10.8 Secure media flow

Files:

- app/Http/Controllers/CourseMediaController.php
- app/Services/CloudinaryPrivateMediaService.php
- resources/views/courses/secure-media-viewer*.blade.php

Flow:

1. User opens secure-view route for a course item.
2. Controller authorizes by role:
   - admins/hr/it can access
   - student must be enrolled
   - trainer must be assigned
3. If item uses Cloudinary metadata:
   - direct URL is generated
   - stream route is prepared
   - download route is prepared for task/quiz files
4. Depending on asset type:
   - video streams inline
   - pdf streams inline
   - docx/pptx streams inline/proxied
   - office docs can use Google viewer URL
   - external resource_url redirects away if no secure asset
5. SecurityHeadersMiddleware adjusts CSP/frame rules for media viewing.


10.9 Submission and review flow

Files:

- app/Http/Controllers/CourseItemSubmissionController.php
- app/Notifications/SubmissionReceivedNotification.php
- app/Notifications/QuizLiveNotification.php

Student submit flow:

1. Student submits task file or quiz answer.
2. Item type must be task or quiz.
3. Student must be enrolled in item's course.
4. Quiz must be live before answer is allowed.
5. Task requires file upload.
6. Quiz requires answer_text.
7. Submission is stored.
8. Progress row is updated to completed.
9. Trainer receives database notification if available.

Review flow:

1. Trainer/admin opens latest submissions.
2. Only latest submission per enrollment/item is reviewable.
3. review_status is updated.
4. review_notes is required for revision_requested.
5. CourseProgress is adjusted:
   - reviewed => completed_at kept/set
   - revision_requested => completed_at cleared

Quiz live flow:

1. Trainer/admin toggles quiz live state.
2. live_at is set when turned on.
3. Assigned students receive database notification.

15. IMPORTANT ARCHITECTURAL NOTES
================================================================

1. The application currently contains both a newer week/session/item course structure and an older day/day-item structure.
   The main routed flow uses weeks/sessions/items.
   The day-based controller and models appear to be legacy support or transition code.

2. Secure course media depends on Cloudinary configuration.
   If Cloudinary credentials are missing, secure upload/update flows will fail.

3. Several features gracefully check whether optional tables exist before using them.
   Examples:
   - notifications
   - activity_logs
   - demo_review_videos
   - position column in demo_feature_videos

4. The codebase relies heavily on Blade templates with large inline styling rather than a thin API + SPA structure.

5. Activity logging is a major cross-cutting concern.
   It is implemented both explicitly and through middleware.

6. Student progress is item-based.
   Opening content can mark progress complete, and reviews can reopen work by clearing completion.

7. Trainer access is enforced through enrollments, not just role alone.

8. Demo flows are isolated from normal student course flows.

6. System creates intro/main/task/quiz items automatically.
7. Admin uploads secure files to items.
8. Admin creates enrollment linking course + student + trainer.
9. Student gets email and database notification.

This is not a simple CRUD app.

- catalog management
- structured learning delivery
- secure media streaming
- trainer review workflow
- certificate generation
- demo experience tooling
- reporting/export tools
- audit visibility
- notification delivery

The most important business backbone is:

User -> Role -> Enrollment -> Course -> Week -> Session -> Item -> Progress -> Submission -> Review -> Certificate

The most important management backbone is:

Admin/Superadmin -> Users/Categories/Courses/Enrollments/Demo Tasks/Broadcasts -> Activity Logs and Panels
